# FORGE Skynet Governance Model

Status: ARCHITECTURE DISCOVERY / NOT RATIFIED

Scope:

- No implementation.
- No engines.
- No schemas.
- No Build Tree modifications.
- No law ratification.

## 1. Governance Purpose

Skynet governs all Forge intelligence systems by defining what they must never
do.

It exists above domains and engines.

Engines may evolve.

Architectures may evolve.

Intelligence domains may evolve.

Skynet Laws remain above them until amended through constitutional process.

## 2. Governance Layers

Recommended conceptual hierarchy:

```text
Forge Constitution
└── Skynet Laws
    ├── Contextual Skynet Policies
    ├── Domain Governance Rules
    ├── Rule Pack Constraints
    └── Engine / Intelligence Outputs
```

Skynet Laws are not implementation modules.

They are constitutional constraints.

## 3. Law Lifecycle

### Candidate

A principle is proposed as a possible Skynet Law.

Requirements:

- source discovery
- cross-domain relevance
- risk if violated
- proposed law wording
- known contextual limitations

### Review

The candidate is tested against:

- universality
- conflicts
- implementation neutrality
- human dignity
- evidence discipline
- future-domain relevance

### Ratification

Ratification requires explicit human constitutional approval.

No intelligence system can self-ratify.

No engine implementation can imply ratification.

### Active Law

Once ratified, every intelligence must obey it.

### Amendment

Amendment requires:

- reason
- impact analysis
- affected domains
- migration notes
- human approval
- versioned record

### Retirement

Retirement should be rare.

It requires proof that the law is obsolete, harmful or superseded by a better
law.

## 4. Conflict Model

When an intelligence conflicts with a Skynet Law, the law wins.

Possible outcomes:

- block output
- degrade output
- request evidence
- request human review
- mark recommendation invalid
- produce a safer explanation
- refuse a judgment

Examples:

If Manager Intelligence wants to rank advisors by a single growth score:

- conflict with "No Single Score Represents Human Growth"
- output should be blocked or redesigned

If Career Intelligence labels someone not ready without evidence:

- conflict with "Evidence Before Judgment"
- output should request missing evidence or human review

If Purpose Snapshot is used to pressure production:

- conflict with "Purpose Is Context, Not Pressure"
- output should be blocked

## 5. Scope of Governance

Skynet governs:

- data interpretation
- classifications
- recommendations
- summaries
- manager-facing outputs
- advisor-facing outputs
- AI explanations
- future intelligence domains

Skynet does not govern:

- every UI style decision
- ordinary file structure
- harmless implementation details
- domain-specific calculations already governed by rule packs unless a law is
  implicated

## 6. Universal Laws and Contextual Policies

Top-level Skynet Laws should be universal.

Contextual rules should be policies below the law layer.

Example:

```text
Skynet Law:
Human Dignity Must Be Preserved

Contextual Policy:
Latent demand must not justify manipulation in financial advisory contexts.
```

This prevents SMNYL or financial advisory context from becoming hardcoded into
Forge Core.

## 7. Ratification Authority

Candidate ratification process:

- proposed in PAQ
- reviewed by human constitutional owner
- documented in constitution file or amendment
- assigned version
- listed as active law

Not valid:

- implicit lock through implementation
- AI-generated law activation
- engine-level override
- Build Tree status alone

## 8. Law Count Governance

Target:

3 to 7 active Skynet Laws initially.

Why:

- fewer laws are easier to remember
- fewer laws reduce bureaucracy
- laws should be foundational, not procedural

If more than 7 laws are proposed, prefer:

- parent law with child policies
- domain governance rule
- rule pack constraint
- implementation acceptance criteria

## 9. Governance Against Bureaucracy

Skynet should avoid bureaucracy by:

- staying compact
- avoiding procedural micromanagement
- governing only high-risk intelligence behavior
- focusing on constraints
- separating laws from implementation details
- using contextual policies for local cases
- treating laws as blockers only for serious conflicts

Skynet should not become a committee for every feature.

Skynet should be a small set of red lines.

## 10. Future Review Questions

Before ratification, Forge should define:

- exact law wording
- law owner
- law versioning
- conflict outcome standards
- human review triggers
- relationship to existing Constitution
- relationship to rule packs and organization profiles
- law amendment process

## 11. Governance Verdict

CANDIDATE FOR LOCK AS GOVERNANCE MODEL.

Reason:

Forge needs a layer that all intelligence systems obey, but the governance model
must remain small, universal and implementation-neutral.

No law is ratified by this document.
