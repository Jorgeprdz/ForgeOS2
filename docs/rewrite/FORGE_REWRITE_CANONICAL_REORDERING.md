# Forge Rewrite Canonical Reordering

Status: `CANONICAL_DEFINITIVE_ARTIFACT_DAG`

The Forge OS V2 rewrite is ordered by architectural artifacts, not by numeric `SG-*` identifiers. Stage IDs remain stable only for governance, receipts, logs, aliases and historical traceability.

## Source Of Truth

The execution source of truth is:

`stage.produces + stage.consumes + material output evidence -> artifact graph -> topological waves -> canonical sequence -> next executable stage`

Not authoritative:

- numeric `SG-*` order;
- physical JSON order;
- hand-written `canonical_order`;
- `.forge/rewrite/completed_stages` without material evidence;
- narrative documentation;
- implicit assumptions in runners.

Processable artifacts:

- `scaffolds/manifest/rewrite-stages.json`
- `scaffolds/manifest/rewrite-artifact-graph.json`
- `scaffolds/manifest/canonical-rewrite-sequence.json`
- `scaffolds/validation/lib/rewrite-artifact-dag.mjs`

## Artifact Model

Each produced artifact is expanded into a graph record with:

- `artifact_id`
- `artifact_type`
- `produced_by`
- `materialization.kind`
- `materialization.paths`
- `materialization.validator`
- `consumed_by`
- `version`
- `status`
- `constitution_refs`
- `owner_decision_refs`
- `receipt_template`

An artifact does not exist merely because it is listed in `produces`. It must be validated by one of:

- current artifact receipt;
- historical compatibility rule;
- generated contract or file;
- stage evidence plus validated state;
- manifest entry validated by scaffold validators.

## Deterministic Ordering

`canonical_order` is generated by `node scaffolds/validation/generate-rewrite-artifact-graph.mjs`.

Tie breaker:

1. constitutional priority;
2. critical-path priority;
3. downstream unblock count;
4. explicit stage priority;
5. stable stage ID.

The generated graph must be reproducible. `validate-rewrite-artifact-graph.mjs` and `test-rewrite-orchestrator.mjs` reject manipulated order, missing producers, duplicate producers, cycles and non-deterministic output.

## Completion Model

Completion is artifact-level.

States:

- stage discovered;
- stage ready;
- stage running;
- artifact produced;
- artifact validated;
- stage partially complete;
- stage complete;
- stage blocked;
- stage failed.

Partial completion is allowed when a stage produces independent artifacts. Consumers unlock only when their consumed artifacts are materialized and valid. A pending decision on one artifact must not block unrelated artifacts from the same stage.

Receipts are generated at runtime, not during this audit. Receipt template:

```json
{
  "artifact_id": "",
  "stage_id": "",
  "version": "",
  "materialized_at": "GENERATED_AT_RUNTIME",
  "evidence": [],
  "validators": [],
  "validation_status": "PASS|FAIL",
  "content_hash": "GENERATED_AT_RUNTIME",
  "constitution_refs": [],
  "decision_refs": []
}
```

Timestamps and hashes are runtime evidence and are not used to derive the deterministic graph.

## SG-021 Decision Isolation

`SG-021 - Product Semantics And Domain Authority Decisions` is not a global blocker.

It produces:

- `ProductSemanticsDecisionRecord`: derived from Constitution, AGENTS and Product Spec.
- `DomainAuthorityDecisionRecord`: defaulted to safe reversible unknown/no-authority semantics.
- `AdvisorOwnerDecisionSet`: pending owner decision, scoped only to Advisor identity, lifecycle, Advisor access, Relationship, NASH and Advisor Workspace consumers.

Decision categories:

- Category A `DERIVED`: repository evidence answers the question.
- Category B `DEFAULTED`: safe reversible default, no side effects, unknown remains unknown.
- Category C `PENDING`: irreductible owner/product meaning or human authority risk.

SG-021 pending decisions block only declared `affected_artifacts` and `blocked_consumers`. Product Catalog does not consume `AdvisorOwnerDecisionSet` and is not blocked by Advisor owner decisions.

## Adding Or Changing Stages

To add a stage:

1. add stable `id`;
2. declare `produces`;
3. declare every `consumes`;
4. declare material output paths and validators;
5. declare owner decisions only where irreductible;
6. run `node scaffolds/validation/generate-rewrite-artifact-graph.mjs`;
7. run `npm run scaffold:validate`.

To change a producer:

1. move the artifact to the new producer;
2. preserve `previous_stage_id` or alias metadata;
3. regenerate the DAG;
4. verify no duplicate producer, missing producer or cycle exists;
5. update migration notes if historical traces change.

Do not sort by `SG-*`. Do not resolve cycles by removing true dependencies. Extract earlier primitives when needed.
