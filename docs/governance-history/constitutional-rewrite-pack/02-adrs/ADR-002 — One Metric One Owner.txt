ADR-002 — ONE METRIC ONE OWNER

1. TITLE

ADR-002 — One Metric One Owner

2. STATUS

Proposed / Draft

3. CONTEXT

ADR-001 estableció cómo Forge reconoce evidencia válida, ownership conceptual de evidencia, fuentes válidas, evidencia parcial y unknown handling.

ADR-002 no redefine evidencia.

ADR-002 gobierna métricas oficiales.

Forge operará sobre múltiples dominios que consumirán métricas relacionadas con:

- producción
- actividad
- persistencia
- compensación
- forecast
- comportamiento
- productividad
- desarrollo
- business planning
- manager intelligence
- economic motivation
- NBA

Sin ownership métrico explícito, Forge puede crear múltiples versiones de la misma métrica, producir cálculos inconsistentes, generar conflictos entre dominios, contaminar Forecast, contaminar Compensation Intelligence, producir recomendaciones contradictorias y degradar trazabilidad.

Diferencia conceptual:

Evidence:
información o fuente que sustenta una afirmación.

Fact:
afirmación confirmada dentro de un contexto.

Metric:
valor definido, calculado o interpretado bajo una regla oficial, periodo, owner y contexto.

ADR-001 protege la validez de la evidencia.

ADR-002 protege la verdad oficial de las métricas.

4. DECISION

Toda métrica importante dentro de Forge debe tener un único owner conceptual responsable de su definición oficial.

Otros dominios pueden:

- consumir la métrica
- visualizar la métrica
- interpretar la métrica
- contextualizar la métrica
- explicar la métrica
- usarla como input para decisiones propias

Pero no pueden:

- recalcularla como verdad paralela
- redefinir su significado
- modificar su fórmula oficial
- producir versiones incompatibles
- ocultar periodo, snapshot o contexto
- usar una versión local como métrica oficial

Una métrica puede tener muchas interpretaciones.

Una métrica no debe tener muchas verdades oficiales.

5. DEFINITIONS

Metric

Valor cuantitativo o estado calculado que representa una dimensión comercial, operativa, conductual, económica o de desempeño dentro de Forge.

Metric Owner

Dominio autorizado para definir la versión oficial de una métrica, incluyendo significado, fórmula, contexto, periodo y límites.

Metric Consumer

Dominio que usa una métrica oficial producida por otro owner para interpretar, explicar, planear, recomendar o contextualizar.

Official Metric

Métrica reconocida como versión oficial dentro de Forge para un contexto definido.

Derived Metric

Métrica calculada a partir de una o más métricas oficiales o facts oficiales.

Snapshot

Versión preservada de reglas, métricas, cálculos o contexto en un momento o periodo específico.

Reporting Period

Periodo al que pertenece una métrica.

Metric Context

Condiciones bajo las cuales una métrica tiene significado: carrier, canal, rule pack, periodo, producto, advisor, manager, equipo, país o contexto operativo.

Metric Interpretation

Lectura, explicación o significado que un dominio produce usando una métrica oficial.

Metric Drift

Deterioro de consistencia cuando una métrica cambia de significado, fórmula o uso sin gobernanza explícita.

Metric Conflict

Situación donde dos fuentes, dominios o cálculos producen valores incompatibles para la misma métrica.

Metric Truth

Versión oficial reconocida de una métrica para un contexto y periodo determinado.

Metric Boundary

Límite que define qué dominio posee la métrica, qué dominios pueden consumirla y qué no puede redefinirse.

6. METRIC OWNERSHIP RULES

Una métrica importante debe tener un owner oficial.

Ownership no depende de:

- UI donde aparece
- engine que la calcula
- dominio que la consume más
- usuario que la consulta
- dashboard donde se muestra
- recomendación que la utiliza

Owner significa:

dominio autorizado para definir la versión oficial de la métrica.

Ejemplos conceptuales:

- Compensation Intelligence owns compensation metric interpretation.
- Productivity owns activity-to-output productivity metrics.
- Mick owns observable behavior pattern metrics.
- Forecast owns scenario metrics, not source facts.
- Business Planning consumes official metrics to build action paths.
- Manager Intelligence consumes metrics to support coaching, not to redefine metric truth.

7. OFFICIAL METRIC RULES

Forge debe mantener una sola verdad oficial por métrica dentro de un contexto definido.

Permitido:

- múltiples visualizaciones
- múltiples explicaciones
- múltiples interpretaciones
- múltiples usos por distintos dominios
- múltiples snapshots históricos
- escenarios derivados claramente etiquetados

Prohibido:

- múltiples cálculos oficiales incompatibles
- redefiniciones locales no gobernadas
- cambios silenciosos de fórmula
- mezcla de periodos
- reinterpretación histórica sin snapshot
- consumer domains creando metric truth paralela

Los snapshots históricos deben preservarse.

Cambios de definición métrica requieren gobernanza explícita y no deben reescribir el pasado como si siempre hubiera usado la nueva definición.

8. CONSUMER DOMAIN RULES

Forecast

Puede:
Consumir métricas oficiales para producir escenarios, riesgos, proyecciones y gaps.

No puede:
Redefinir producción, compensación, productividad, persistencia o facts oficiales.

Forecast owns scenarios, not facts.

Business Planning

Puede:
Consumir métricas oficiales para construir rutas de acción, prioridades y planes.

No puede:
Crear métricas paralelas para justificar planes.

Manager Intelligence

Puede:
Consumir métricas oficiales para apoyar coaching, observación y desarrollo.

No puede:
Redefinir desempeño, compensación, productividad o conducta como truth paralela.

Productivity

Puede:
Poseer métricas de actividad productiva, conversión y throughput dentro de su boundary.

No puede:
Redefinir compensación, forecast, behavior truth o relación humana.

Compensation Intelligence

Puede:
Poseer interpretación oficial de métricas de compensación bajo rule packs, periodos y snapshots.

No puede:
Redefinir métricas productivas o forecast como hechos.

Mick

Puede:
Poseer métricas o patrones de comportamiento observable: hábitos, consistencia, cadence, completion, activity generation.

No puede:
Redefinir productividad, compensación, intención interna, motivación o valor humano.

Advisor Experience

Puede:
Consumir métricas oficiales y señales de uso para adaptar ayuda, progresión y aprendizaje.

No puede:
Redefinir métricas comerciales, comportamiento, productividad o compensation truth.

Economic Motivation

Puede:
Consumir métricas económicas oficiales, compensation context y forecast scenarios claramente etiquetados.

No puede:
Redefinir ingresos, comisiones, forecast o producción para motivar acciones.

NBA

Puede:
Consumir métricas oficiales y escenarios para priorizar acciones.

No puede:
Redefinir métrica, crear cálculo propio oculto o convertir escenario en fact.

9. DERIVED METRICS

Una métrica derivada es válida cuando:

- usa métricas oficiales o facts oficiales
- declara inputs
- declara owner
- declara contexto
- declara periodo
- declara límites
- no se presenta como métrica original
- no contradice la métrica oficial

Una derivación deja de ser interpretación y se convierte en nueva métrica cuando:

- tiene fórmula propia
- será reutilizada por otros dominios
- influye recomendaciones de forma independiente
- se reporta como valor estable
- se compara entre periodos, advisors, managers o equipos
- cambia decisiones fuera del contexto local

Cuando una derivación se convierte en nueva métrica, necesita owner.

Para evitar verdades paralelas:

- consumer domains deben etiquetar derivaciones
- no deben ocultar inputs
- no deben usar nombres ambiguos
- no deben presentar cálculos locales como oficiales

10. METRIC CONFLICT HANDLING

Cuando dos fuentes producen resultados distintos:

- Forge debe marcar conflicto.
- Forge no debe elegir silenciosamente una verdad.
- Forge debe identificar fuentes, periodos, owners y contexto.
- Forge debe conservar uncertainty hasta resolución.
- Forge puede recomendar validación.

Cuando una métrica cambia de definición:

- el cambio debe ser explícito
- el periodo efectivo debe declararse
- snapshots previos deben preservarse
- comparaciones históricas deben advertir cambio de definición

Cuando existe conflicto de periodo o snapshot:

- Forge no debe mezclar periodos incompatibles
- Forge no debe reinterpretar pasado con reglas futuras
- Forge no debe combinar rule packs sin boundary explícito

11. SNAPSHOT AND PERIOD RULES

Toda métrica sensible al tiempo debe preservar:

- periodo efectivo
- snapshot aplicable
- contexto de regla
- fecha o ventana de cálculo
- versión de definición cuando aplique

Forge no debe:

- mezclar periodos incompatibles
- recalcular historia con reglas futuras como si fueran actuales
- comparar métricas con definiciones distintas sin advertencia
- ocultar que una métrica pertenece a un snapshot histórico

Historical truth must remain historical truth.

Un análisis posterior puede existir, pero debe etiquetarse como análisis posterior, no como reemplazo del histórico.

12. PROHIBITIONS

Forge no debe:

- crear múltiples verdades métricas
- redefinir métricas desde consumer domains
- mezclar snapshots incompatibles
- recalcular métricas oficiales arbitrariamente
- usar métricas obsoletas como actuales
- ocultar cambios de definición
- permitir metric drift
- presentar derivaciones locales como métricas oficiales
- mezclar forecast con fact metrics
- mezclar compensation metrics con productivity metrics
- mezclar behavior metrics con human worth
- usar métricas sin owner conceptual cuando el impacto sea relevante
- permitir que NBA, NASH, Forecast o Manager Intelligence inventen métricas para justificar recomendaciones

13. CONSEQUENCES

Positive Consequences

Este ADR habilita:

- consistencia métrica
- trazabilidad
- menor conflicto entre dominios
- estabilidad de Forecast
- estabilidad de Compensation Intelligence
- estabilidad de Business Planning
- estabilidad de Manager Intelligence
- recomendaciones menos contradictorias
- mejor gobernanza de snapshots históricos
- menor drift arquitectónico
- base clara para Productivity Metric Ownership
- base clara para métricas económicas y de compensación

Tradeoffs

Este ADR introduce:

- más gobernanza
- más disciplina arquitectónica
- menos flexibilidad para cálculos locales
- mayor dependencia de ownership claro
- más trabajo al crear métricas derivadas
- más fricción cuando un dominio quiere calcular rápidamente algo nuevo
- necesidad de resolver conflictos antes de conclusiones fuertes

Estos tradeoffs son aceptables porque Forge no puede operar como Decision Intelligence confiable si sus métricas tienen múltiples verdades paralelas.

17. FINAL VERDICT

ADR-002 establece que Forge no puede mantener Decision Intelligence consistente sin ownership métrico explícito.

Toda métrica importante debe tener un owner conceptual responsable de su definición oficial.

Otros dominios pueden consumir, visualizar, interpretar, contextualizar y explicar métricas oficiales, pero no pueden redefinirlas como verdades paralelas.

ADR-001 protege la verdad de la evidencia.

ADR-002 protege la verdad de las métricas.

Juntos constituyen la base de gobernanza sobre la cual pueden construirse Forecast, Compensation Intelligence, Productivity, Manager Intelligence, Business Planning, NBA y cualquier dominio futuro sin generar múltiples verdades paralelas.
